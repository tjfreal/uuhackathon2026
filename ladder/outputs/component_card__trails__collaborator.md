# Trails
*Sensemaking layer · proposal*

It makes temporal reconstruction a first-class tool, which raises design questions about granularity and evidence.

This is proposal, not a finished tool. It matters here because it shows where the system is trying to go next.

**What it is**: Reconstructs how an idea developed across time and notes.

**The problem it solves**: Once an idea is spread across daily notes, project files, and revisions, it becomes hard to see the shape of its development.

**A concrete example**: The idea of the library as a ladder did not begin with AI or note-system tooling; Trails would help show how that service idea later became a design principle for Ladder.

**How it fits**:
- Depends on the same note history that note-system preserves.
- Can use notes surfaced by Surface or connected by Bridge.
- Supports Ladder's need to explain lineage rather than just features.

**Inputs**:
- local note corpus
- timestamps
- links between notes
- repeated concepts across entries

**Outputs**:
- an ordered trail showing how an idea evolved
- candidate milestones in a line of thought

**Dependencies**:
- note-system corpus
- note links
- time metadata

**Current limitations**:
- It is still a proposal rather than a finished tool
- Time order does not automatically explain causation
- Good trails need enough note density to reconstruct something real

**Next rung for you**: Define the minimum evidence needed for a trail to feel trustworthy rather than narratively convenient.
